# CURB — native iOS push (APNs) implementation plan

> Generated by the design-curb-native-push workflow (2026-06-15). The iOS app is a WKWebView wrapper; this adds native APNs alongside the existing Web Push so the move-your-car alerts work natively. Gated on the active Apple Developer membership.

## Approach

Add a parallel native-APNs transport that reuses CURB's existing forever-watch pipeline end-to-end: same spot/rule shape, same lead-window + night-before math, same dedupe (notifiedFor/notifiedEveFor), same forever-watch re-arm, same CRON_SECRET-gated cron and the same GitHub Action. Web Push is untouched. Native iOS device tokens live in a NEW Upstash hash curb:apns (keyed by hex token) so they never have to satisfy save-subscription's validSubscription() (which requires a web-push https endpoint + p256dh/auth keys). A new POST /api/save-ios-subscription validates the token + sanitizes the spot (reusing the existing sanitizeSpot/sanitizeRule, factored into a shared module), and send-notifications.js gains a second loop that POSTs to APNs over node:http2 with a cached ES256 JWT, mirroring the web-push branch's window/dedupe/410-prune semantics. On the client, the SwiftUI WKWebView wrapper gets an AppDelegate (via @UIApplicationDelegateAdaptor) + a curbPush bridge that mirrors the existing curbLocation/curbShare pattern; the web's existing "Sweep alerts" button is diverted to the native bridge before the Add-to-Home-Screen hint fires.

## Endpoints

- POST /api/save-ios-subscription (NEW): accepts { token:hex, platform:'ios', bundleId:'guide.curb.ios', spot }; validates the hex token (rejects via the same path the web validator rejects non-push endpoints), sanitizes spot via the shared sanitizeSpot/sanitizeRule, stores in curb:apns; returns { ok, stored }.
- GET/POST /api/send-notifications (CHANGED): unchanged auth (CRON_SECRET Bearer) and web-push loop; adds a second loop that delivers due APNs pushes over one http2 session per run with a cached ES256 JWT, reusing the same re-arm/window/dedupe logic and pruning curb:apns records on 410/400. Also adds /b/<cnn> deep-link url to both web and APNs payloads.
- POST /api/save-subscription (CHANGED, behavior-preserving): sanitizeSpot/sanitizeRule extracted to api/_spot.js and re-imported; no functional change to web-push.
- POST /api/enable-auto-park, POST /api/parked (UNCHANGED): remain web-only; documented as the place native auto-park would later hook in via a token-hash credential.

## Backend changes

- api/_store.js: add a parallel hash TKEY_IOS='curb:apns' (field = hex device token, value = { token, spot, notifiedFor, notifiedEveFor, savedAt, platform:'ios', bundleId }). Add helpers mirroring the web-push ones but token-keyed: saveIosSub(token, spot) (same notifiedFor/notifiedEveFor preservation on same-nextSweepISO re-save, same carry-forward of rule/cnn/sideKey, stamps savedAt), advanceIosSpot(token, newSpot) (resets dedupe, preserves savedAt), loadAllIosSubs() (returns [{token, spot, notifiedFor, notifiedEveFor, savedAt}]), deleteIosSub(token) (called on 410 Unregistered / 400 BadDeviceToken), markIosNotified(token, nextSweepISO, field). Keep these next to the existing saveSub/advanceSpot/loadAllSubs/deleteSub/markNotified so the shapes stay identical.
- api/_spot.js (NEW shared helper, '_' prefix = not routed): export sanitizeSpot and sanitizeRule, moved verbatim out of save-subscription.js (they currently live as module-private functions there). Re-import them in save-subscription.js so web-push behavior is byte-identical, and import them in the new save-ios-subscription.js so the cron sees the EXACT same spot/rule shape (3-char weekday label, clamped leadMinutes, eveningISO-before-sweep guard, digits-only cnn, sideKey<=8). Do NOT copy-paste — single source of truth.
- api/save-ios-subscription.js (NEW endpoint): POST { token, platform:'ios', bundleId, spot }. Validate token = /^[0-9a-fA-F]{64,200}$/ (APNs hex token, length-bounded; NOT validSubscription()). Reject if bundleId present and !== APNS_BUNDLE_ID. cleanSpot = sanitizeSpot(spot); 400 if null. storeReady() guard. saveIosSub(token, cleanSpot). Return { ok:true, stored:true }. No ownerProof needed (auto-park is web-only today); if native auto-park is ever wanted, gate it with a token-hash credential, never anything the cron reads.
- api/_apns.js (NEW shared helper): an APNs sender over node:http2 + node:crypto, no new heavy deps (mirrors the hand-rolled style of the repo). Exports: getProviderToken() — mints an ES256 JWT { header:{alg:'ES256',kid:APNS_KEY_ID}, payload:{iss:APNS_TEAM_ID, iat:nowSec} } signed with APNS_KEY_P8 (run .replace(/\\n/g,'\n') in case newlines were escaped), base64URL-encoded (no +/=), CACHED in a module-scope var and re-minted only when iat age > ~50 min (APNs 429s a re-mint <20 min apart and 403s an iat >1h). openSession()/sendOne(session, jwt, token, payload) — POST :path /3/device/<token> with headers authorization: bearer <jwt>, apns-topic: APNS_BUNDLE_ID, apns-push-type: alert, apns-priority: 10, apns-collapse-id: <tag>; resolve { status, reason }. Host from APNS_HOST (default api.push.apple.com; sandbox api.development.push.apple.com). Caller MUST session.close() at end of run so the function event loop can exit.
- api/send-notifications.js: after the existing web-push loop, add a SECOND loop over loadAllIosSubs() that reuses the IDENTICAL forever-watch re-arm pass (recomputeSpot + advanceIosSpot + continue), the IDENTICAL lead/delta math, and the IDENTICAL night-before + 30-min-lead window selection and per-window dedupe (notifiedEveFor/notifiedFor). Differences only: (a) build an aps payload { aps:{ alert:{title,body}, sound:'default', 'thread-id':tag }, url, tag } instead of the web JSON; (b) deliver via _apns.sendOne over a single http2 session opened once for the whole run; (c) on status 410 (Unregistered) or 400 (BadDeviceToken) -> deleteIosSub(token) + pruned++ (the APNs analogue of the web 410/404 prune); (d) markIosNotified outside the send try/catch. session.close() in a finally. Return ok with both channels' counts. ALSO: set url to '/b/'+spot.cnn (falling back to '/') in BOTH the web and iOS payloads so a notification tap opens the right block — cnn is already in the stored spot; this is the one shared web+native payload change.
- api/parked.js: NO change required now (auto-park is web/Shortcut-only). Note for later: if native auto-park is added, it would resolve a token-hash to a curb:apns record and call saveIosSub instead of saveSub.
- package.json: no new dependency if hand-rolling _apns.js with node:http2 + node:crypto (recommended, matches the repo's web-push-minimal style). If preferring a library, add 'apns2' (native http2, mints+caches the JWT, surfaces APNs reason codes) and call client.close() per invocation.
- .env.example: add APNS_KEY_P8 (full multi-line .p8 PEM), APNS_KEY_ID (10-char), APNS_TEAM_ID (10-char = 69K2S3664V), APNS_BUNDLE_ID=guide.curb.ios, and optional APNS_HOST (defaults to production). Document that .p8 newlines must survive (store raw multiline in Vercel UI, or escaped \n + replace before crypto).
- vercel.json: optional — add a functions entry for api/send-notifications.js with maxDuration (e.g. 60) if the combined web+APNs batch risks the default timeout. No 'crons' key change needed; the GitHub Action remains the real trigger and drives APNs for free.
- .github/workflows/sweep-alerts-cron.yml: NO change — the same curl to /api/send-notifications with Bearer CRON_SECRET now fans out to both transports.

## Store design

Two sibling Upstash hashes, identical record shape, different key + transport. EXISTING curb:subs (untouched): field = web-push subscription.endpoint -> { subscription:{endpoint,keys:{p256dh,auth}}, spot, notifiedFor, notifiedEveFor, savedAt, proofHash? }. NEW curb:apns: field = hex APNs device token -> { token, spot, notifiedFor, notifiedEveFor, savedAt, platform:'ios', bundleId }. The spot/rule/eveningISO sub-shape is byte-identical across both (same sanitizeSpot/sanitizeRule), so recomputeSpot() and the entire forever-watch re-arm + lead-window + night-before + per-window dedupe logic apply unchanged to both — only the load/advance/markNotified/delete helpers differ by hash, and only the delivery transport differs (webpush.sendNotification vs http2 POST). They coexist as fully independent records: a user on BOTH the PWA and the native app has one record in each hash and gets pushed on both channels (cross-device de-dup is out of scope — there is no shared identity key, which is acceptable). No web-push field (p256dh/auth/endpoint) ever appears in curb:apns, so save-subscription's validSubscription() never has to accept a non-web token, and the hex token never has to masquerade as a subscription object. ownerProof/auto-park tokens (curb:tokens) stay web-only and untouched.

## Risks / footguns

- Sandbox vs production host footgun: TestFlight uses PRODUCTION APNs, only Xcode debug builds use sandbox. A token from the wrong environment returns 400 BadDeviceToken. Drive host from APNS_HOST; consider a one-time cross-host retry on 400 before pruning so an env-mismatched token isn't deleted permanently.
- JWT lifecycle is load-bearing: must cache the ES256 provider token and re-mint only every ~50 min — minting more than once per ~20 min returns 429 TooManyProviderTokenUpdates, and an iat >1h old returns 403 ExpiredProviderToken. Module-scope cache works on warm Fluid Compute instances but a cold start re-mints; that's fine (well within the 20-min floor at a 15-min cron) — just never re-mint per-token.
- http2 session lifecycle: open ONE session per cron invocation, multiplex all due iOS pushes, then session.close() in a finally. Leaving it open keeps the event loop alive and the function may hang to timeout (billing + missed runs). Do not rely on a session surviving between 15-min ticks — guard session.closed/destroyed and reconnect lazily, but correctness must not depend on reuse.
- Combined batch may exceed the function timeout once both channels run — set maxDuration in vercel.json for send-notifications if the sub count grows; bound work per run.
- Prune semantics differ from web: APNs uses 410 Unregistered / 400 BadDeviceToken instead of HTTP 410/404. Map both to deleteIosSub, and keep markIosNotified OUTSIDE the send try/catch (same as web) so a transient store error isn't read as a send failure and re-pushed next tick.
- Deep-link payload change touches BOTH channels: adding /b/<cnn> to the web payload changes existing web-push behavior (currently url:'/'). The SW notificationclick already navigates to data.url, so /b/<cnn> will route web users to the block page too — verify that's desired (it is consistent), and that /b/:cnn rewrite (vercel.json) renders correctly.
- Entitlement promotion: manual/CI archives can ship aps-environment=development, which makes the production host reject the token. Verify the archived entitlement is 'production'.
- .p8 newline corruption: if APNS_KEY_P8 is stored with escaped \n, the ES256 sign will fail with an opaque crypto error. Normalize with .replace(/\\n/g,'\n') before importing the key.
- Cross-channel double-notify: a user registered on both PWA and native gets two notifications. Acceptable per scope (no shared identity), but call it out to Alejandro.
- sanitizeSpot/sanitizeRule extraction must be behavior-identical — re-import in save-subscription.js rather than fork, or web-push and native could silently diverge in what counts as a valid forever-watch rule.

## Effort & build order

Roughly 2-3 focused days. Suggested build order: (1) Apple account setup — create the .p8 APNs key, enable Push on guide.curb.ios, set the four APNS_* Vercel env vars (unblocks all server testing). (2) Backend store + shared spot helper — extract api/_spot.js, add curb:apns helpers in _store.js (no behavior change to web path; easy to unit-test the dedupe/carry-forward against the existing saveSub). (3) api/save-ios-subscription.js — stand it up and POST a fake hex token + a real spot, confirm it lands in curb:apns with the right shape. (4) api/_apns.js + the send-notifications.js second loop — test against a real device token from a dev build hitting the sandbox host; verify JWT caching, 200 delivery, and 410/400 pruning; add the /b/<cnn> payload change. (5) iOS client — entitlement + pbxproj wiring, AppDelegate + PushRouter, the curbPush bridge in ContentView, and the 1-line onAlertTap divert in index.html; test permission prompt -> token POST -> real push -> tap opens /b/<cnn>, including cold-start deep link. (6) Archive to TestFlight and re-test against the PRODUCTION host (the most failure-prone step — env/entitlement mismatches surface here). Web Push remains fully functional throughout since every change is additive or behavior-preserving.
